## 4.1 现实地基:汉字拆解是已解决的数据问题

汉字"由什么组成、按什么结构组成",不需要从零定义。已有现成标准:

- **IDS(Ideographic Description Sequences,表意文字描述序列)**:用一组**结构描述符(IDC)**——⿰(左右)、⿱(上下)、⿲(左中右)、⿳(上中下)、⿴(包围)等十余个——精确描述一个字的组成。
  - 例:明 = ⿰日月;森 = ⿳木木木;焚 = ⿱(⿰木木)火。
- **开源数据集**:CHISE 项目的 IDS 库、GitHub 的 cjkvi-ids、官方《汉字部件规范 GF 0014-2009》、Unihan 数据库(含读音)。
- **字音字义**:CC-CEDICT、Unihan 等开源词典。

### 关键认知

这些数据是**给语言学家的**,它会一路递归拆到笔画级,大量"部件"既不可玩也不被普通人认识。

> 因此引擎**不是"直接用 IDS"**,而是"**以 IDS 为原料,人工策划出一套适合玩的子集**"。原始数据是起点,不是成品。

## 4.2 数据模型

引擎中有三类对象。

### 4.2.1 部件(Component)

原子级可玩单位。人工从 IDS 叶子中精选约 **100~200 个**游戏相关部件(并非 IDS 的全部叶子)。

| 字段 | 说明 |
|---|---|
| id | 唯一标识 |
| 字形 | 部件的字形(火、日、月、氵…) |
| 读音 / 名称 | 玩家可见 |
| 属性 | 火 / 水 / 木 / 金 / 土 / 心 / 无 |
| 稀有度 | 影响掉落与商店出现率 |
| 单独出战效果 | **关键**:部件可单独打出的弱效果(防卡手用,见 4.4) |

### 4.2.2 汉字(Character)

可出战的字卡。

| 字段 | 说明 |
|---|---|
| id | 唯一标识 |
| 字形 | 汉字字形 |
| 拼音 / 释义 | 玩家点击可见(认字门槛缓解 + 寓教于乐) |
| 属性 | 由主部首判定 |
| 阶级 | 基础 / 进阶 / 高阶 |
| 战斗效果 + 数值 | 出字时触发的技能(手工设计,见 4.6) |
| 稀有度 | — |
| **配方** | 该字由哪些原料组成,是引擎最核心的字段 |

### 4.2.3 配方图谱(Recipe Graph)

所有汉字的配方连成一张**有向无环图(DAG)**:

- **叶子节点** = 部件(原料来源)。
- **非叶节点** = 汉字,其配方指向自己的原料。
- 原料**可以是部件,也可以是更低阶的汉字**。
- 图中无环:配方永远"向下"指向更原子的对象(焚 需要 林,林 需要 木,不会反过来)。

这张 DAG 就是玩家的「字之科技树」。

## 4.3 核心设计原则:游戏配方 ≠ 语言学拆解

这是整个引擎最重要的一条认知。

「焚」在语言学上 = 木 + 木 + 火(完全展开)。但在游戏里,它的配方可以被定义为:

- 方案 A:焚 = 木 + 木 + 火(一步合成)
- 方案 B:焚 = 林 + 火(两步合成,需先合「林」)

**策划选择把配方定义在哪一层。** 配方定得浅 → 合成快、节奏轻;配方定得深(多步)→ 合成像解谜、高阶字"来之不易"。

> **配方深度,就是玩法节奏的调参旋钮。** 通常:基础字配方浅,高阶字配方深,让强力的字需要被"层层构筑"出来。

> **已拍板(2026-06-07,Q1 关闭)**:整体粒度采用**方案 A(一步合成)**——浏览器 demo 做过 A/B 双模式实测,一步合成的节奏更爽、决策更聚焦。高阶字仍可按需使用多步配方制造"来之不易"感,但默认从浅。

## 4.4 操作规则

### 4.4.1 拆(Dismantle)

- 输入:字库中的一个汉字。
- 行为:把该字**配方中的全部原料**(下一层)放入部件池。
- 例:拆「焚」→ 林 + 火;此后「林」还可再拆 → 木 + 木。
- 独体字 / 部件不可拆(它们是图的叶子)。
- 基础规则:**无损返还**。损耗或加成由法宝调整(如「偏旁部首表」拆字额外 +1 部件)。

### 4.4.2 合(Compose)

- 输入:部件池中的部件,以及字库中的低阶字(4.2.3:原料可以是更低阶的汉字)。
- 行为:若部件池 + 字库合计包含某汉字配方的**全部原料**,即可锻造该字,放入字库,消耗对应原料。**原料优先取部件池,池中没有再消耗字库中的低阶字**(3.9 战例「合林 → 合焚」即此语义;v0.4 澄清)。
- 只有"正字"(真实的、被设计过的汉字)可被锻造满威力(见 4.5)。

---

## 4.5 三层"真字"系统

引擎最难的设计抉择:什么样的部件组合"算一个字"?

两个失败方向:

- **太严**(只允许凑出真实常用字):组合空间稀疏,玩家常手握部件却合不出东西 → **卡手**。
- **太松**(任意组合都能合出"某个字"):丢掉"这是真汉字"的灵魂 → 变成抽象拼图。

解法是**三层系统**:

| 层级 | 定义 | 强度 |
|---|---|---|
| 1. 正字 | 配方产出真实的、被策划设计过的汉字 | 满威力("已验证"层) |
| 2. 部件直出 | 任何部件**与无战斗效果的材料字**永远可单独打出一个弱效果(未定义效果者用统一「兜底一击」:单体 3 伤,走生克)| 弱(**防卡手地板,永不 brick**;v0.5 实测修正:兜底覆盖字库与部件池全部对象) |
| 3. 讹字(可选层) | 不成字的部件硬凑,合出"讹字" | 虚弱、带负面 |

**第三层的妙处**:乱凑字合出虚弱的讹字,在剧情上正好说得通——乱拼字就是在替反派「讹」干活。**游戏世界观自己解释了"为什么乱拼不好"。**

> 核心原则:在"什么算正字、给满威力"上**从严**;用"部件直出"**兜底**,保证玩家永远有得打。

---

## 4.6 战斗属性层

- **属性**:由汉字的主部首**自动判定**(火/灬→火系,氵→水系……)。
- **具体效果与数值**:**必须手工逐字设计**。"焚 打 18 点 AOE"是平衡决策,无法从部首推导。部首决定"家族",策划决定"招式"。
- **例外**:升阶链(火→炎→焱→燚)这种"同部首、量递增"的结构子图,可半程序化定义其数值曲线。

---

## 4.7 提示引擎(防卡手关键系统)

拆/合机制最大的体验风险是"我不知道能拼出什么"。提示引擎解决它。

**功能**:给定玩家当前部件池,实时计算并展示——

1. **可合成**:哪些字的配方已被池中部件完全满足 → 直接显示可锻造的字。
2. **差一步提示**:哪些字"还差一个部件即可合成" → 显示"还差一个『火』即可合『焚』"。

**实现**:在配方 DAG 上做查询。按"原料"给所有配方建立倒排索引;部件池变化时增量重算被影响的配方节点。

**附加价值**:

- 同时驱动新手引导(教学时直接高亮可合成项)。
- 驱动「推敲」法宝(预览下回合可合成的字)。
- 本身是一个潜移默化的**识字工具**。

---

## 4.8 边界情况与设计旋钮

| 情况 | 处理 |
|---|---|
| 独体字 | 既是部件也是可出战的字,是图的叶子,不可再拆 |
| 一料多用 | 同一部件(如「木」)被多个配方共享 → DAG 天然支持,即共享原料 |
| 多解 | 某组部件可合成多个字时,提示引擎列出全部,玩家选择 |
| 配方深度 | 按字 / 按稀有度调整;高阶字配方更深,使其"来之不易" |
| 数据规模 | 首发约 200 部件 + 300~500 个设计过的字;全 3500 常用字是长线内容储备 |

## 4.9 内容生产管线

引擎不能"导入 IDS 即用"。从原始数据到一套可发布的、好玩的字 / 部件 / 配方,中间是一条完整的内容生产管线。本节详述这条管线。

### 4.9.0 总览:管线分两半

整条管线可以清晰地切成两半,这个区分决定了人力该投在哪:

- **可自动化的一半**:数据采集、字集初筛、属性自动判定、图谱校验、配置导出。这些是确定性的脚本工作,是 Claude Code 的主场。
- **依赖判断的一半**:部件库精选、游戏配方的粒度撰写、战斗效果与数值设计、平衡迭代。这些是真正的"游戏设计",必须人来做。

> 管线设计的核心原则:**在能自动化的地方狠狠自动化,把人力集中到判断密集的步骤上。** 把整条管线当成纯数据工作,或当成纯手工工作,都是错的。

管线共九个阶段。

### 4.9.1 阶段一 · 数据采集

汇集三类开源数据,合成一张"原始母表"。

| 数据 | 来源 | 用途 |
|---|---|---|
| 拆解结构(IDS) | CHISE IDS 库 / GitHub cjkvi-ids | 每个字的拆解树 |
| 字集 | 《通用规范汉字表》一级字表(3500 字) | 候选字范围 |
| 拼音 | Unihan(kMandarin)/ pypinyin 等 | 玩家可见读音 |
| 释义 | CC-CEDICT / Unihan(kDefinition) | 玩家可见释义 |
| 部首 | Unihan(kRSUnicode)/ 康熙部首表 | 属性自动判定的依据 |

**产出**:一张原始母表,每行一个字,含字形、IDS 拆解、拼音、释义、部首。

### 4.9.2 阶段二 · 字集筛选

3500 一级字不会、也不需要全部做成可玩字。MVP 只需约 150~200 个设计过的字,长线目标约 500,3500 是"字典图鉴"的终极收集目标。本阶段从 3500 中圈出"候选字池"。

筛选标准:

- 能拆解出**可辨识**的部件(拆出来全是怪部件的字会劝退玩家)。
- 归属于一个有用的**属性家族**(火/水/木/金/土/心)。
- **常用、认得**(生僻字进不了战斗用字,但可留作图鉴收集)。
- 有清晰的**战斗幻想**(这个字"看起来像"什么效果)。

**产出**:候选字池。初筛可由脚本按部首、字频粗筛;精筛由人工定夺。

### 4.9.3 阶段三 · 部件库定义

从 IDS 的叶子中,精选约 100~200 个游戏部件。

- 每个部件填:字形、读音、属性 tag、稀有度、**单独出战效果**(防卡手用)。
- 区分**带属性部件**(火/水/木/金/土/心)与**中性部件**(无)。

> **关键:部件库与字集是"共同设计"的。** 不能先定全部部件再定全部字,也不能反过来——两者互相约束(部件集必须能拼出字集)。正确做法是**按属性家族成组推进**:一次锁定一个属性家族的字与它需要的部件,让这一块的配方图谱"闭合"。

### 4.9.4 阶段四 · 游戏配方撰写(管线的硬核)

为每个候选字撰写它的**游戏配方**。

- 这一步**不是数据录入,是游戏设计**——每条配方都是一个关于节奏的小决策(见 4.3)。
- 配方原料可以是部件,也可以是更低阶的字。
- 粒度规则:基础字配方浅(一步),高阶字配方深(多步),让强字"层层构筑"出来。
- 升阶链(火→炎→焱→燚)按结构化模板撰写。
- 必须保证图谱是 DAG——配方永远指向更原子的对象,不成环。

这是整条管线**最耗人力、最吃判断**的一步,无法外包给脚本。

### 4.9.5 阶段五 · 属性与战斗效果

- **属性**:由主部首自动判定(脚本完成,人工抽查例外)。
- **战斗效果 + 数值**:逐字**手工设计**——"焚 打 18 点 AOE"是平衡决策,推导不出来。效果要与该字所属流派的玩法定位呼应(见第 5 章)。
- 升阶链的数值曲线可半程序化生成,再人工微调。

### 4.9.6 阶段六 · 图谱校验(自动化 QA)

配方图谱必须有一套**自动校验脚本**,在每次内容改动后运行(纳入 CI)。校验项:

| 校验项 | 说明 |
|---|---|
| 无环 | 图谱必须是 DAG |
| 原料存在 | 每个配方引用的部件 / 字都已定义 |
| 无孤儿部件 | 每个部件至少被一条配方使用(否则标记) |
| 可达性 | 每个字在一局中确实能被凑出来,未被不可获取的部件锁死 |
| 防卡手覆盖 | 任意合理的部件池都至少能合出或直出点东西 |
| 升阶单调 | 升阶链的数值随阶级单调递增 |
| 分布健全 | 属性分布、AP 消耗分布无明显畸形;无某字严格碾压另一字 |

### 4.9.7 阶段七 · 平衡迭代

- 阶段五的数值是**首版基准**,必经实测调整(见第 10 章平衡哲学)。
- 全部配方、效果、数值以**配置表**形式存在,改平衡**不需要改代码**。
- 软启动后接入遥测:流派胜率、字的选取率、卡点数据,数据驱动调参。

### 4.9.8 阶段八 · 工具链

支撑这条管线,需要几件工具:

- **数据采集脚本**(Python):摄取 IDS + 字表 + 拼音 + 释义 → 原始母表。
- **配方表 + 校验脚本**:配方可用电子表格维护,校验脚本(阶段六)对其做静态检查。
- **配置导出**:把表导出为游戏读取的 JSON。
- **图谱可视化**:把配方 DAG 画出来,帮策划直观看到"字之科技树"。
- 校验脚本接入 CI,内容一改即自动检查。

### 4.9.9 阶段九 · 分批节奏

内容不是一次产完,按**垂直切片**分批——一次做完一个属性家族(字 + 部件 + 配方 + 效果),保证那条流派立刻可玩。

| 批次 | 内容范围 |
|---|---|
| MVP | ~200 字 + ~100 部件,3 个流派家族(火 / 水 / 金)+ 土系工具字(10~15)+ 少量木字(供克制与跨界配方) |
| 上线后扩展 | 补齐 5 属性,扩至 ~500 字 |
| 长线 | 逐步向 3500 常用字推进(字典图鉴收集目标) |

### 4.9.10 工作量定性与 Claude Code 分工

| 阶段 | 性质 | 承担 |
|---|---|---|
| 一 数据采集 | 自动化 | Sonnet 写采集脚本 |
| 二 字集筛选 | 半自动 | 脚本初筛,人工精筛 |
| 三 部件库定义 | 判断 | 人工(与阶段四共同设计) |
| 四 配方撰写 | **判断(硬核)** | 人工为主;Sonnet 可按既定规则批量填充常规条目 |
| 五 属性与效果 | 属性自动 / 效果判断 | 属性脚本判定;效果数值人工设计 |
| 六 图谱校验 | 自动化 | Opus 定校验算法,Sonnet 实现脚本 |
| 七 平衡迭代 | 判断 | 人工 + 遥测数据 |
| 八 工具链 | 自动化 | Sonnet 实现,Opus 定数据结构 |

> 一句话:管线的"数据骨架"(阶段一、六、八)交给 Claude Code 高效产出;管线的"设计灵魂"(阶段三、四、五、七)必须人来注入。

---

## 4.10 Claude Code 分工

| 角色 | 职责 |
|---|---|
| Opus | 数据模型设计、提示引擎算法、拆/合运算与正字校验的核心逻辑 |
| Sonnet | 配方图谱与字表的批量装配(从 IDS 源)、配置读取、字段填充 |
| 人工 | 逐字撰写游戏配方与战斗效果、数值平衡迭代 |

配方图谱是**数据**(JSON 配置表);引擎逻辑是**纯 C# 算法**,无需任何美术或编辑器操作,是 Claude Code 最擅长的部分。

---

## 4.11 小结

这套引擎立住后,五行、流派、法宝、字怪弱点全都只是挂在配方图谱上的**规则层**。它是《字·斗》真正的地基,也是最该优先在纸面原型与战斗原型阶段验证的系统。
