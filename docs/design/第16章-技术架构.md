## 16.1 引擎选型(已收口:Unity)

> **最终决策(2026-07-04):Unity(C#)**。Q12 关闭。以下候选对比保留作为决策依据存档。

目标平台变更后,引擎选型理由变了。「一套代码导出微信小游戏」不再是约束,引擎候选重新摆开:

| 候选 | Steam(PC/Mac) | iOS/Android | TypeScript | 个人开发者生态 | 评价 |
|---|---|---|---|---|---|
| **Cocos Creator 3.x** | ✅(Native build) | ✅ | ✅ | 国内强,Steam 案例少 | TypeScript 对 AI 协作友好;Steam 上不是主流(社区资源少) |
| **Unity** | ✅(Steam 主流) | ✅ | ❌(C#) | 全球最强,Steam 80%+ 占有率 | Steam 默认选择,C# 与 Claude Code 协作没问题;但学习曲线 |
| **Godot 4** | ✅(GDScript / C# / Native) | ✅(实验性 iOS,Android 稳定) | ❌ | 增长迅速,独立游戏圈友好 | 完全免费、开源、无版权费;iOS 导出仍偶有坑 |
| **Defold** | ✅ | ✅ | ❌(Lua) | 小众,卡牌品类有案例 | 轻量、跨端稳;社区相对小 |

**选择 Unity 的理由**:

1. **Steam 是首发主战场**:Unity 在 Steam 占有率 80%+,Steamworks 集成、第三方插件、社区资源、疑难排查案例都最丰富,是 Steam 独立游戏的默认选择。
2. **C# 与 Claude Code 协作顺畅**:拆合引擎、战斗结算、地图生成等纯逻辑部分用 C# 实现无障碍,TypeScript 的 AI 协作优势并非不可替代。
3. **移动端导出成熟**:iOS / Android 构建管线经过大量商业项目验证,比 Godot(iOS 导出偶有坑)与 Cocos(Steam 案例少)风险更低。

**落地要求**:锁定 Unity LTS 版本;阶段 0 用 1~2 天跑通最小验证——空项目导出 Windows 可执行 + Steamworks SDK 集成(成就/Cloud)。

## 16.2 平台适配层(v0.3 简化)

三个平台(Steam / iOS / Android)的 API 差异仍在,但**比四端小游戏简单得多**:

| 能力 | Steam | iOS | Android |
|---|---|---|---|
| 存档 | Steam Cloud(可选)+ 本地 | iCloud(可选)+ 本地 | Google Play Saved Games(可选)+ 本地 |
| 成就 | Steamworks Achievements | Game Center(可选) | Google Play Games(可选) |
| 排行榜 | Steamworks Leaderboards(可选) | Game Center(可选) | Google Play Games(可选) |
| 分享/截图 | Steam 自带 F12 | 系统截图 | 系统截图 |
| 输入 | 键鼠 + 手柄 + Steam Input | 触屏 | 触屏 |

抽象一个 `PlatformSDK` 接口,把上述能力统一封装。**单机游戏不需要"账号"、"支付"、"广告" 这三大类**——这就是 v0.3 比 v0.2 简化的核心。

## 16.3 核心数据:配方图谱

- 拆合规则引擎的配方图谱(见第 4 章)以 **JSON 配置表**形式存在。
- 以开源 IDS 数据集为原料,人工策划装配。
- 配方、属性、战斗效果、数值均为数据驱动,便于调整与扩展。
- **Premium 模式下数据全部打包在客户端**,无服务端校验需求,无热更新依赖。

## 16.4 包体管理(v0.3)

| 端 | 约束 |
|---|---|
| Steam(PC/Mac) | **无硬约束**。1~2 GB 都正常,玩家不在意 |
| iOS / Android | App Store/Play 仍有 200MB 起的"是否走 Wi-Fi 下载"门槛,**建议控制在 200MB 以内**;包体优化是加分项不是硬约束 |

"字即画"的美术方向与"无帧动画"的动效方向天然有利于控制包体——移动端 200MB 完全做得到。Steam 端可以追加高清字体包 / 4K 背景图等增量素材。

## 16.5 项目规范:CLAUDE.md

在项目根目录写好 `CLAUDE.md`,记录:

- 引擎版本与目录规范。
- 三端 API 差异与适配层约定。
- 代码风格、命名规范。
- 配方表 / 字表的数据结构定义。
- **本作的核心架构原则**:无服务端、无账号、无内购、纯单机。

这是与 Claude Code 高质量协作的地基——它让每次代码生成都遵循同一套约定。

## 16.6 Claude Code 分工

| 角色 | 职责 |
|---|---|
| **Opus**(想清楚的事) | 整体架构、拆合引擎数据模型、提示算法、战斗结算与地图生成的核心逻辑、平台适配层设计、跨端疑难排查 |
| **Sonnet**(量大的事) | 配方表与字表的批量装配、UI 组件、配置读取、关卡数据填充、单元测试、文档 |
| **人工** | 引擎编辑器内的场景搭建、UI 布局、美术资源接入;数值平衡的迭代调试;Steam 商店页素材准备 |

实操上:用 Opus 先把架构与接口定义敲定、生成清晰的模块边界,再切到 Sonnet 按接口批量填充实现。

**关键认知**:拆合引擎、战斗结算、地图生成、提示算法全是纯逻辑——这是 Claude Code 加速最猛的部分;而场景搭建、UI 布局、美术、Steam 商店素材 Claude Code 做不了,需在编辑器和外部工具里手工完成。

## 16.7 发布策略:跨端就绪,顺序发布

- 引擎一套代码导出三端,**"跨端就绪"从第一天成立**。
- **发布顺序**:**Steam 优先,移动端随后或同步**——Steam 是 indie 卡牌的主战场,商店页 / Wishlist / 测试机制都最完善;移动端在 Steam 上线 1~3 个月后同步推出,可吸收一波 Steam 口碑红利。
- 不再有"先国内单端试水"的概念。

## 16.8 服务端:可选,不是刚需(v0.3 大幅简化)

> **v0.5 修订(2026-07-05)**:养成 RPG 转向后,「零服务端」结论**部分推翻**——计时开箱、每日刷新、每日宝箱上限要求**可信时间源**(第 19 章 19.9),需要一个极轻量服务端(单一校时端点 + 可选存档校验);另需接入广告 SDK 与订阅计费(StoreKit / Play Billing)。除此之外(存档/成就/排行榜)仍按本节「平台代办」原则,不自建。

v0.3 回退到 Premium 买断制后,服务端从 v0.2 的「刚需」回退到「几乎不需要」。

**可以完全不做服务端**:

- 存档:本地存档 + 平台云存档(Steam Cloud / iCloud / Google Play Saved Games)
- 成就:全部由平台 SDK 承载(Steamworks / Game Center / Google Play Games)
- 设置:本地配置文件

**可选的轻量服务端能力**(若做,优先级排序):

| 能力 | 是否做 | 实现方式 |
|---|---|---|
| 全球排行榜(每日挑战 / 段位段终时长) | **可选,推荐做** | Steamworks Leaderboards(免费、平台原生);移动端走 Game Center / Play Games |
| 玩家匿名遥测(留存、流派分布、卡点等) | 可选 | 单一 HTTP endpoint + 一份 SQLite/Postgres;或用 Aptabase / PostHog 等托管方案 |
| 反馈收集 | 可选 | 一个公开邮箱 / Discord 频道即可,不需要服务端 |
| 跨设备云存档 | 平台代办 | Steam Cloud / iCloud / Google Play —— **完全不用自建** |

> **零服务端的好处**:0 运维成本、0 月度账单、0 数据库维护、0 服务可用性焦虑、0 安全责任。对个人开发者是巨大的减负。

## 16.9 与其他系统的关系

- **拆合引擎(第 4 章)**:配方图谱是核心数据,引擎逻辑是 Claude Code 主战场。
- **美术 / 音频(第 12、13 章)**:无帧动画、音频压缩的选择服务于包体控制(移动端)与跨端一致性。
- **变现(第 14 章 v0.3)**:Premium 买断 → 无支付集成、无广告集成、无内购恢复机制;只需对接平台的"购买验证 API"(Steam 默认通过、iOS/Play 自动通过)。
- **留存(第 15 章)**:DLC 与版本更新是留存抓手;不再依赖服务端的赛季 / 战令机制。
- **MVP(第 17 章 v0.3)**:发布顺序与 MVP 的阶段验证一致;无服务端能力分阶段开放的复杂度。
